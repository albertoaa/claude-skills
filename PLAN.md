 Plan: claude-skills Library Repository

 Context

 The user wants /Users/aaa/workspace/personal/claude-skills to become a curated, portable library of Claude Code slash commands. The repo should be consumable as a git submodule in any project, with a
 lightweight script for enabling/disabling individual skills per project. The core problem is that skill files must physically exist in .claude/commands/ to be discovered by Claude Code — the
 management script bridges the library and the consuming project via symlinks.

 ---
 Repository Structure

 claude-skills/
 ├── LICENSE
 ├── README.md           # setup instructions for consuming projects
 ├── CATALOG.md          # human-readable index of all skills
 ├── skills.sh           # management script (enable/disable/list/status/restore)
 ├── skills/             # slash command skill files (one .md per skill)
 │   ├── code-review.md
 │   ├── commit.md
 │   └── ...
 └── behaviors/          # CLAUDE.md snippet files (for auto-invoked behaviors)
     └── ...

 Why flat skills/*.md over SKILL.md plugin format: The commands/*.md format is directly discovered by Claude Code via filesystem. It requires no plugin system. Each skill is one file = one symlink =
 one /slash-command.

 Why behaviors/ is separate: Claude Code cannot auto-invoke behaviors from .claude/commands/. Automatic behaviors belong in CLAUDE.md via @path references. This makes the distinction explicit rather
 than hidden.

 ---
 Skill File Format

 All skills in skills/ follow the Claude Code command format:

 ---
 description: Short description shown in /help
 argument-hint: <optional-hint>
 allowed-tools:
   - Bash(git *:*)
   - Read
 ---

 ## Your task

 ...instructions for Claude...

 The filename (without .md) becomes the slash command name: skills/code-review.md → /code-review.

 ---
 skills.sh Interface

 Run from the consuming project root, pointing at the submodule path:

 ./vendor/claude-skills/skills.sh enable code-review commit   # symlink into .claude/commands/
 ./vendor/claude-skills/skills.sh disable code-review         # remove symlink
 ./vendor/claude-skills/skills.sh list                        # list all available skills
 ./vendor/claude-skills/skills.sh status                      # show which are enabled
 ./vendor/claude-skills/skills.sh restore                     # re-create symlinks from .claude/skills.json

 The script:
 1. Creates/removes symlinks in .claude/commands/
 2. Updates .claude/skills.json (derived artifact tracking enabled list)
 3. Uses jq if available, falls back to grep parsing for the JSON
 4. restore reads skills.json and recreates all symlinks (for fresh clones)

 ---
 .claude/skills.json (in consuming project)

 {
   "source": "vendor/claude-skills",
   "enabled": ["code-review", "commit"]
 }

 Checked into version control. Regenerated by enable/disable. Symlinks are the ground truth; this is for readability and restore.

 ---
 Consuming Project Workflow

 Initial setup:
 git submodule add https://github.com/albertoaa/claude-skills vendor/claude-skills
 git submodule update --init
 ./vendor/claude-skills/skills.sh enable code-review commit

 Fresh clone (teammate):
 git submodule update --init
 ./vendor/claude-skills/skills.sh restore

 Updating library:
 cd vendor/claude-skills && git pull && cd -
 git add vendor/claude-skills && git commit -m "update claude-skills"

 Using a behavior (in project's CLAUDE.md):
 @vendor/claude-skills/behaviors/code-style.md
